### 핵심 발견사항 (Top 5)

1. **보고서 데이터 휘발성** — DocEditorPanel의 문서 데이터가 `useState`에만 저장되어, 페이지 이동이나 새로고침 시 작업 내용이 손실될 가능성이 있음
2. **보안 구조 취약** — 비밀번호 평문 저장, API 키 하드코딩, 클라이언트 사이드 권한 체크만 존재하여 보안 개선이 가장 시급
3. **모바일 하단 네비게이션 밀집** — 8개 메뉴가 360px 이하 화면에서 라벨이 잘리며 사용성 저하
4. **URL 라우팅 미지원** — SPA임에도 `history.pushState`를 사용하지 않아 브라우저 뒤로가기 및 딥링크 불가
5. **단일 파일 구조의 한계** — 약 9,500줄 단일 HTML 파일로 배포는 간편하나, Babel Standalone 런타임 컴파일로 초기 로드가 느리고 장기 유지보수에 불리

### 2.2 Major (핵심 기능 영향)

#### BUG-M01: 보고서(DocEditorPanel) 데이터 휘발 가능성

| 항목 | 내용 |
|------|------|
| **심각도** | Major |
| **영역** | 보고서(Document Editor) |
| **현상** | DocEditorPanel의 `allDocs` 상태가 `useState`로만 관리되어, 페이지 이동이나 브라우저 새로고침 시 작성 중인 보고서 데이터가 손실될 가능성 있음 |
| **원인** | `usePersistedState`(localStorage 연동) 대신 일반 `useState` 사용 여부 확인 필요 |
| **영향** | 사용자가 장시간 작성한 보고서가 예고 없이 사라질 수 있음 |
| **권장 조치** | `allDocs` 상태를 `usePersistedState` 또는 별도 localStorage 저장 로직으로 전환, 자동 저장 주기 설정(30초 간격 등) |

#### BUG-M02: PDF 출력 시 팝업 차단기에 의한 차단

| 항목 | 내용 |
|------|------|
| **심각도** | Major |
| **영역** | 견적서 PDF 출력 |
| **현상** | `window.open()`을 사용한 PDF 출력 시, 브라우저 팝업 차단기가 활성화된 경우 PDF 창이 열리지 않음 |
| **원인** | `window.open()` 호출이 직접적인 사용자 클릭 이벤트 핸들러 내에서 실행되지 않거나, 비동기 처리 후 호출되는 경우 팝업 차단 대상이 됨 |
| **영향** | 팝업 차단기를 사용하는 사용자(대다수)가 PDF 출력 기능을 사용할 수 없음 |
| **권장 조치** | `iframe` 기반 인쇄 방식 또는 `Blob URL` 생성 후 다운로드 방식으로 대체 구현 |

#### BUG-M03: 모든 보안이 클라이언트 사이드에서만 수행

| 항목 | 내용 |
|------|------|
| **심각도** | Major |
| **영역** | 인증/인가 전반 |
| **현상** | API 키, 비밀번호 등이 클라이언트 코드에 노출되어 있으며, 권한 체크가 프론트엔드에서만 수행됨 |
| **원인** | 서버 사이드 인증/인가 레이어 미구현 |
| **영향** | 브라우저 개발자 도구로 권한 우회 가능, API 키 탈취 위험 |
| **권장 조치** | 서버 사이드 인증 레이어 추가 (JWT 기반), API 키를 환경변수로 분리, 민감 정보 코드 제거 |

### 2.3 Minor (부분 기능 영향)

#### BUG-m01: 대시보드 수익률 100.0% 표시 오류

| 항목 | 내용 |
|------|------|
| **심각도** | Minor |
| **영역** | 대시보드 |
| **현상** | 수익률이 100.0%로 표시되나, 실제 견적 합계(41,947,400원)가 계약금액(42,000,000원)보다 낮아 정확한 수익률 계산 방식에 의문 |
| **원인** | 수익률 계산 공식이 `(계약금액 - 지출) / 계약금액`이 아닌 다른 기준을 사용하거나, 지출 데이터가 0으로 처리됨 |
| **권장 조치** | 수익률 계산 공식 확인 및 실제 견적/지출 데이터 기반으로 정확한 수치 표시 |

#### BUG-m02: 모바일 하단 네비게이션 밀집

| 항목 | 내용 |
|------|------|
| **심각도** | Minor |
| **영역** | 모바일 UI |
| **현상** | 하단 네비게이션에 8개 메뉴가 배치되어, 360px 이하 화면 너비에서 라벨이 잘리거나 아이콘이 밀집됨 |
| **원인** | 메뉴 항목 수가 모바일 하단 네비에 적합한 4~5개를 초과 |
| **권장 조치** | 주요 5개 메뉴 + "더보기" 버튼으로 재구성 |

#### BUG-m03: 간트차트 진행률 모두 0%

| 항목 | 내용 |
|------|------|
| **심각도** | Minor |
| **영역** | 간트차트 |
| **현상** | 간트차트 내 모든 항목의 진행률이 0%로 표시됨 |
| **원인** | 세부항목(subtask)이 없거나, 체크리스트 가져오기 후 진행률 연동이 되지 않음 |
| **권장 조치** | 체크리스트 완료 상태와 간트차트 진행률을 실시간 연동하는 로직 추가 |

#### BUG-m04: 동기화 상태바 모바일 텍스트 오버플로

| 항목 | 내용 |
|------|------|
| **심각도** | Minor |
| **영역** | 동기화 상태 표시 |
| **현상** | 모바일 화면에서 동기화 상태바의 텍스트("오후 06:54 D1+Drive...")가 길게 표시되어 잘림 |
| **원인** | 텍스트 오버플로 처리(ellipsis) 미적용 |
| **권장 조치** | `text-overflow: ellipsis` 적용 또는 모바일에서 축약 텍스트 사용 |

## 5. 성능 이슈

### 5.1 단일 HTML 파일 번들 크기

- **현상:** 약 9,500줄의 단일 HTML 파일에 모든 컴포넌트, 스타일, 로직이 포함
- **영향:** 초기 로드 시 전체 코드를 다운로드 및 파싱해야 하므로 로드 시간 증가
- **권장 조치:** 코드 스플리팅(Code Splitting) 도입, Vite 또는 Next.js 빌드 시스템 전환 검토

### 5.2 전체 데이터 일괄 로드

- **현상:** 모든 프로젝트 데이터를 앱 시작 시 한 번에 로드
- **영향:** 프로젝트 수가 증가할수록 초기 로드 시간 및 메모리 사용량 증가
- **권장 조치:** 페이지네이션 또는 가상 스크롤(Virtualization) 적용, 필요한 데이터만 지연 로딩(Lazy Loading)

### 5.3 localStorage 용량 제한

- **현상:** localStorage의 5MB 제한으로 대규모 데이터 저장 시 용량 초과 가능
- **영향:** 프로젝트 및 문서가 많아지면 저장 실패 발생 가능
- **비고:** Cloudflare D1 동기화로 이미 보완되고 있으나, 오프라인 시 로컬 데이터 관리 방안 필요
- **권장 조치:** IndexedDB 활용 검토 (용량 제한 거의 없음), localStorage 사용량 모니터링 로직 추가

### 5.4 Babel Standalone 런타임 컴파일

- **현상:** React Production 빌드를 사용하고 있으나, Babel Standalone으로 브라우저에서 JSX를 런타임 컴파일
- **영향:** 초기 로드 시 Babel 파싱 및 트랜스파일 오버헤드 발생, 저사양 기기에서 체감 지연
- **권장 조치:** 사전 빌드(Pre-build) 방식으로 전환하여 런타임 컴파일 제거

## 6. 보안 이슈

### 6.1 비밀번호 평문 저장

- **위험도:** 높음
- **현상:** 사용자 비밀번호가 평문(plain text)으로 localStorage 및 클라우드(D1/Drive)에 저장
- **영향:** 브라우저 개발자 도구나 클라우드 데이터 접근 시 비밀번호 즉시 노출
- **권장 조치:** bcrypt 또는 SHA-256 해시 적용, 서버 사이드 인증 시 salt 추가

### 6.2 GAS URL 하드코딩

- **위험도:** 중간
- **현상:** Google Apps Script URL이 소스 코드에 직접 포함
- **영향:** 코드 공개 시 GAS 엔드포인트 노출, 악용 가능
- **권장 조치:** 환경변수 또는 별도 설정 파일로 분리, 접근 제한(IP 화이트리스트 등) 추가

### 6.3 Cloudflare Worker URL 하드코딩

- **위험도:** 중간
- **현상:** Cloudflare Worker URL이 소스 코드에 직접 포함
- **영향:** Worker 엔드포인트 노출로 무단 접근 가능
- **권장 조치:** API 키 기반 인증 추가, Worker에서 origin 검증

### 6.4 클라이언트 사이드 권한 체크

- **위험도:** 높음
- **현상:** 관리자/일반 사용자 구분이 프론트엔드 코드에서만 수행
- **영향:** 브라우저 개발자 도구로 권한 값 변조 시 관리자 기능 접근 가능
- **권장 조치:** 모든 권한 민감 API에 서버 사이드 검증 추가

### 6.5 XSS 위험

- **위험도:** 중간
- **현상:** 사용자 입력이 `dangerouslySetInnerHTML` 또는 직접 DOM 삽입으로 처리되는 곳 존재 가능
- **영향:** 악의적 스크립트 주입 시 세션 탈취 등 보안 사고 발생 가능
- **권장 조치:** 모든 사용자 입력에 대한 sanitize 처리, `dangerouslySetInnerHTML` 사용 최소화, DOMPurify 라이브러리 도입 검토

## 8. 결론 및 권고사항

### 8.1 전반적 평가

EventOS v1.0은 이벤트 기획에 필요한 핵심 기능들(프로젝트 관리, 견적서, 체크리스트, 간트차트, 인건비/지출 관리 등)이 안정적으로 동작하고 있습니다. 15개 기능 영역에서 약 92%의 통과율을 기록하였으며, 서비스 중단 수준의 Critical 버그는 없는 상태입니다.

### 8.2 긍정적 요소

- **3계층 동기화 아키텍처** (localStorage / Cloudflare D1 / Google Drive)는 데이터 안정성을 보장하는 좋은 설계로 평가됩니다.
- **단일 HTML 파일 배포**는 인프라 복잡도 없이 즉시 배포가 가능한 장점이 있습니다.
- **프로젝트별 탭 구성**이 직관적이며, 체크리스트-간트차트 연동 등 기능 간 연결이 잘 되어 있습니다.

### 8.3 개선 우선순위

1. **보안 (가장 시급)** — 비밀번호 해시, 서버 사이드 인증, API 키 분리는 서비스 운영 전 반드시 해결해야 할 사항입니다.
2. **데이터 안정성** — 보고서 데이터 영속화(BUG-M01)는 사용자 데이터 손실을 방지하기 위해 즉시 확인이 필요합니다.
3. **UX 개선** — URL 라우팅, 모바일 하단 네비 최적화, 오프라인 안내 등은 사용자 만족도에 직접 영향을 미칩니다.
4. **성능 및 아키텍처** — 빌드 시스템 전환(Vite/Next.js)은 장기적으로 유지보수성과 성능 모두를 개선할 수 있는 핵심 과제입니다.

### 8.4 장기 권고

- **6개월 내** 빌드 시스템(Vite 또는 Next.js) 전환을 권장합니다. 이를 통해 코드 스플리팅, 런타임 컴파일 제거, HMR(Hot Module Replacement) 등의 이점을 얻을 수 있습니다.
- **서버 사이드 인증**(JWT 기반) 도입 시 기존 Cloudflare Worker를 활용하면 추가 서버 없이도 구현 가능합니다.
- **PWA 전환**을 통해 모바일 앱 수준의 경험과 오프라인 지원을 동시에 확보할 수 있습니다.
